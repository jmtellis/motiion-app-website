## Build Philosophy

1. **Each phase ships something usable.** No phase leaves the app broken. After any phase, you can demo what you've built.
2. **Infrastructure before features.** Phase 0 sets up Supabase schema/RLS, auth, design tokens, and error tracking so feature work is fast and clean.
3. **Magic moment first.** The two-sided magic moment (talent sees matched opportunities; industry searches → roster → Project) is reachable by the end of the core MVP phases. Everything else builds on top.
4. **Test as you go.** Each task includes a verification step. Don't defer testing.
5. **Progressive enhancement.** Ship the simplest working version, then layer polish and motion.
6. **Review before proceeding.** Push each completed phase as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review it before starting the next phase.
7. **One schema, two surfaces.** Web and iOS share one Supabase backend. Every schema change is a migration in `supabase/migrations/`; never fork a table for a single surface.
8. **Reconcile, don't rebuild.** This repo already has partial implementations (casting composer, talent navigator, analytics). Extend and reconcile existing code via migrations rather than greenfield rebuilds.

> **Design tokens:** `docs/design.md` exists (source of truth) with a live mirror at `docs/design.html` — dark-first, Geist + Geist Mono, white primary CTA, teal accent (`#2DD4BF`). All styling must reference these tokens.

## Agent Session Guide

### How to Use This Roadmap with Your Coding Agent

1. **Start a session:** Give your coding agent the phase prompt at the beginning of each phase.
2. **Read selectively:** Each phase lists its Reference sections — the specific parts of the PRD, vision, and design docs needed. Read only those, not the entire documents.
3. **Let it work:** The agent reads the roadmap, finds the first unchecked task, implements it, and marks it complete.
4. **One session = one phase (ideally):** Complete a full phase per session for continuity; resume from the last unchecked task if you stop.
5. **Push a PR for review:** When a phase is complete, push the work as a PR and let a review agent (e.g. [CodeRabbit](https://coderabbit.ai)) review before starting the next phase.
6. **Need more context?** If a task references a section not in the phase's Reference sections, read just that section on demand.

### Session Tips

- **Don't read everything:** Each phase's Reference sections tell the agent exactly what to read. Loading full documents wastes context.
- **Don't skip tasks:** Tasks are ordered intentionally; skipping creates dependency issues.
- **Verify after each phase:** Run the app after each phase to confirm everything works.
- **Review before moving on:** Push a PR per phase; don't start the next until it's merged.
- **Update the status line:** After completing tasks, update `**Status:** X/72 tasks complete` and `**Current Phase:** Phase N`.

### Phase Review Workflow

Create a branch per phase (`phase-{N}/{slug}`), commit, push, and open a PR titled `Phase {N}: {Title}` with the phase goal, completed task count, and what to verify. Let an automated review agent ([CodeRabbit](https://coderabbit.ai) recommended) review before merging. Address feedback in follow-up commits; merge only when clean. If you don't use GitHub PR review, per-task verification still applies.

### Prompt Templates

**Starting a new phase:**
> "Read docs/product-roadmap.md and find the current phase. Read only the Reference sections listed for that phase from docs/prd.md, docs/product-vision.md, and docs/design.md. Start working on the first unchecked task. After completing each task, update the checkbox to [x] in the roadmap file. Continue through the phase."

**Resuming after a break:**
> "Read docs/product-roadmap.md. Find where we left off (first unchecked task). Read only the Reference sections listed for the current phase. Continue from the first unchecked task."

**After completing a phase:**
> "Phase [N] is complete. Create a branch called phase-{N}/{slug}, commit all work, push, and open a PR targeting main. Title it 'Phase {N}: {Title}' and include the phase goal and completed task count in the body."

**Fixing an issue:**
> "There's a problem with [description]. Read the relevant section of docs/prd.md for the expected behavior and fix it. Don't mark any new tasks complete until the fix is verified."
